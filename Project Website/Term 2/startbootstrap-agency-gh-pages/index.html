About

Capita is planning to make their SIMS software, a database management system for schools, into a Software as a Service on the cloud. Our project consists of creating a web version of their SIMS Discover app, a data visualisation tool for SIMS. Using a variety of web-based languages and data visualisation libraries, such as D3.js, we hope to recreate SIMS Discover so it can be used by multiple schools online.

Features

Visualisations

A range of visualisations such as bar graphs, line graphs, Venn diagrams, pie charts and donut charts, with tables for more information.

Set Operations

Easily merging and reducing data sets allows you to easily to find the exact information you are looking for.

Easy to Use

The application uses drag and drop from a toolbar listing the data sets and automatically generates the visualisation.

Problem Statement

With Capita's SIMS software planned to become a Software as a Service (SasS), Capita is looking to make both SIMS and SIMS Discover web-based so it can be used through the cloud.

Capita SIMS provides information management systems for schools across the country. One of the products they provide is called SIMS Discover, where teachers can analyse student data and present it in Venn diagrams, charts and graphs to enable them to improve staff methods and identify problems so that they can intervene early.

Currently Discover is a desktop application. SIMS would now like to re-create Discover in the cloud. We need to research how this can be done by combining javascript libraries and then produce a proof of concept that will eventually be taken over by the SIMS Discover team.

Requirements Gathering

Our requirements gathering process consisted of a series of meetings with our client. Initially, the meetings were about the reasons why they created SIMS and what future direction they had in mind for the software. In later meetings, we discussed more in detail the specific requirements for the project, such as having to be a web application. Since our project is to recreate an existing application, most of the requirements were developed after a demo on the SIMS Discover desktop application.

System Architecture

How is our application structured?

Work allocation between our team and team 2

Our team is working on the following components:

  • User Interface
  • Graph Engine
  • Graph Library
  • Set Engine (with Team 2)
The other components are done by Team 2.

Architectural Components

User Interface: Toolbar and canvas which users use to see the visualised data.

Graph Engine: Uses the standard code for a type of diagram and accesses the data file to create a visualisation.

Graph Library: The code used to generate a diagram of the following types: line chart, bar chart, Venn diagram, pie chart and donut chart.

Set Engine: The back end code to merge and query different sets of data, done both by our team and Team 2.

Set Definition Library: The pre-defined sets of data created by Team 2 which our application links to to get data.

Data Sources: The data from the SIMS school data management application.

Design Patterns

Model-View-Controller (MVC)

Our web app uses the MVC pattern, with Model representing the data we use for the visualisations, View being the visualisations created depending on user input, and Controller the algorithm that handles user input and translates it into data visualisations by accessing the Model.

Iterator

The iterator pattern is used to iterate through a set of data and extract the data required for the visualisation that the user wants.

Strategy

The strategy pattern is used in our web application to determine which part of the code has to be run depending on user input.

Prototype

The prototype design pattern is used when merging two visualisations, as it copies the existing data and creates a new visualisation based on it and another set of data.

Interceptor

The interceptor pattern is used for the log in of users and to return them to the canvas once they have logged in.

Implementation Details

Toolbar

The toolbar was implemented using Bootstrap. This framework was particularly useful due to the lack of previous technical experience that we had with web technologies, as it provides with a variety of templates. To achieve consistency throughout the pages of the application, the toolbar was written in the "layout.html" file, which was then imported to every other page in the app by using Flask.

Visualisation

The visualisations were implemented using the library D3.js. Two functions were created to draw the visualisations:

  • drawData: Draws bar charts
  • drawData2: Draws pie charts

There were corresponding functions to get the data for each type of graph: getData and getData2. The data was obtained from the JSON file using jQuery.

Back End

In addition to using Flask for the consistency between pages, we also used it to store user credentials for the log in and register pages due to the simplicity of its Jinja2 templating feature.

Unit and Integration Testing

Unit Testing

For testing the integrity of the components of any software, unit tests have to be done so all the components work while not worrying about how it affects the other functionalities. For our web app, we used local servers to run the individual source codes with multiple entries to validate its workability. The test cases are given below:

Integration Testing

After the unit tests, all the parts were brought together to connect all the functionality to see if it all worked alongside each other and was responsive with respect to its design and interactivity. Being a web application for data visualization, it was critical that all the units integrated and worked properly. We used a Flask interface and server to run all the code together and test its integrity. The only problem that occurred was the ability to connect to Team 2’s server. While working with D3, we came across the problem in connecting to the server that stores the data for the graph. Other than this all the tests cases given in unit tests ran accordingly.

Compatibility Testing

For compatibility, we had to ensure that our web application works on the popular web browsers used. Some web applications run differently based on the browser it is running for, especially visualization libraries such as D3.js. After testing on Internet Explorer, Mozilla Firefox, Safari, Google Chrome, Microsoft Edge, and also on previous versions to check for backward compatibility, we can concluded that our web application is compatible.

User Acceptance Testing

The final phase of Software Engineering involved providing the test cases and prototypes to the users of the app. For our app, it is based on the desktop app SIMS Discover which we used to base our design on. For our final UAT we met with our client to show the functionalities integration of our web app. He has given us good feedback, to be able to fulfill the requirements that he asked from us and stated that he has received what expected the project outcome to be.

Web Technologies

With the project aiming to essentially transform the desktop app discovery SIMS into a cloud based webapp, our client suggested a couple of libraries and languages to be used to experiment and research on so we looked into Lynda.com videos, w3schools and other tutorials to learn more about them and see if they can be implemented on our web app. It is to run on all existing web browsers available.

CSS

This is used for styling and the layout of web pages. CSS is used alongside HTML which constitutes the design and contents of most modern web pages.

JavaScript

Often abbreviated as JS, JavaScript is a dynamic and high-level scripting language. It is an interpreted programming language and is used to add functionality to graphs.

Power BI

Power BI is an interactive data visualization tool we looked into to be able to utilize in our web app. It is a Microsoft Product and can visualize attractive diagrams, but is a standalone program which could not be integrated into another web app.

Phaser IO

This was a Gaming Library that we experimented and looked at to implement interactivity and draggable functions in the app.

jQuery

JQuery is a JavaScript library which makes client side and front-end scripting a lot easier and simpler. It is generally used to add animation, have event handlers and etc.

JSON

JavaScript Object Notation, or JSON for shot is a data structuring format. It is minimal and readable and most modern day technologies are able to interpret the data. JSON is usually used to transmit data between a server and a webapp.


From all these technologies, we used JavaScript, CSS, jQuery, and JSON in our final app design based on the requirements.

Data Visualisation Technologies

Since the main of our Project was data visualization, we looked at a few graphing libraries

Flot

Flot is one of the oldest charting libraries around and focuses on simple usage and interactive features. It’s specifically for jQuery so one needs to be familiar with it before using Flot. This library lets you have full control over the presentation, animation and user interaction.

chart.js

Charts.js is the a very simple JavaScript library and provides clean and easy to implement charts. It is perfect for small Projects and uses HTML5 canvas element for rendering charts. However as per our client’s requirement, the library does not fulfill the types of charts required and all the functionalities to be implemented.

D3.js

Nowadays D3 is often the first name the comes to mind regarding any kind of visualisation. We dwelled deep into its documentation and experimented to see that it’s a very powerful library that lets us create all types of visual effects and chart types. D3 also provides the functionalities and data manipulation techniques required by our client, also being extremely fast, supporting large files and has dynamic behavior.


After experimenting with the ones mentioned above and working through the requirements given by our clients, we came to the conclusion that D3.js will be the best option to use among the graphing libraries, and we have been able to implement them perfectly according to the requirements.

Project Evaluation

Architecture Design

Our architecture was designed using our client's suggested architecture, which was based on their existing desktop application. Our architecture was structured, with the toolbar depending on one file and the visualisations or graph engine depending on another file. We believe it can be easy to understand for other developers. However, we could have divided the graph engine and graph library into two files, with the graph engine only being concerned with processing data and importing specific graph types from the graph library.

User Interface and User Experience

Due to the nature of our project being based on an already existing application, we did not have much creative freedom when designing our user interface. We followed the drag and drop functionality of SIMS Discover, which makes the application intuitive and easy to use. However, the icons only toolbar could be difficult to understand for new users who are not familiar with the meaning of each icon. A possible improvement would have been to include text next to each icon, but this was not a feature of the desktop version.

Functionality

Our application provides the basic functionalities of the original SIMS Discover application, as it has a variety of data visualisations. However, it lacks its in-depth querying of data, which would have been facilitated if we had been able to connect to Team 2s API.

Compatibility

Being a web application, it was essential for the app to be compatible with the most popular web browsers. After performing compatibility testing, we confirmed it was compatible with the top 4 web browsers: Google Chrome, Mozilla Firefox, Safari and Internet Explorer

Maintability

The application is maintainable, as each graphing function follows a similar structure that could be replicated for new types of visualisations. The code is self-explanatory, as it uses basic functionalities from D3.js and the main web languages. In order for developers to add data querying functionalities, they should have a separate back end that will directly provide the final results of queries as JSON files. However, we could have created a detailed developer manual to enable others to easily maintain the application.

Testing

We performed the main types of testing - unit testing, integration testing, compatibility testing and user acceptance testing - to assess our application. The testing process provided a good idea on the functionalities of the application and bugs. To improve our testing process, we should have carried out user acceptance testing with a large number of users, instead of only our client. However, this proved difficult due to our target users being primary and high school teachers.

Project Management

Our project management was reasonable, as we chose on a series of supporting tools to manage the different parts of our project, such as GitHub for version control, Facebook for team communication and Google Drive to store and edit group reporting deliverables. However, our lack of technical experience led to many delays in the development process. To improve this, we should have predicted possible problems and set group deadlines to ensure we were up to date throughout the entire project.
